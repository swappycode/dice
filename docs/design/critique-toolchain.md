## Numbered risks

**1. [BLOCKER] The three docs specify three incompatible wire contracts.** Doc A: `u32 big-endian` length prefix, 256 KiB cap, single `Frame` oneof, message-send over the gateway socket, auth REST bodies in protobuf, typing over QUIC datagrams. Doc B: `u32-LE`, 64 KiB cap, `ClientFrame`/`ServerFrame`, send over socket, auth REST in **JSON**, typing on the control stream. Doc C: `u32 LE`, 1 MiB inbound cap, send over **HTTPS REST** (which Doc B explicitly does not implement â€” the demo client literally could not send a message), datagrams unused. Resume token sizes (16 vs 32 bytes) and resume windows (60 vs 90 s) also differ. **Fix:** before any coding, freeze Doc A's `docs/protocol.md` as the single normative spec and force-reconcile: one endianness (pick LE â€” two of three docs and the client already chose it, and there is no interop reason for BE), 256 KiB cap, sends over the gateway socket with nonce/Ack (Doc C's REST-send section is deleted; its reserved frame number becomes the real path), auth REST bodies protobuf (Doc B's JSON exception dropped), typing on the control stream for M1 (datagrams deferred, see #14).

**2. [BLOCKER] Dev-cert trust chain is internally contradictory and would make the client unable to connect.** Doc A generates **ephemeral in-process self-signed certs per server boot**; Doc C's client (correctly) refuses verification-off and trusts only a **persistent dev CA PEM from disk** (`DICE_EXTRA_CA`), and Doc B needs a CA-signed leaf for the "works in WSS, fails in QUIC" reason it itself flags. Ephemeral per-boot certs mean there is never a CA file for the client to trust. **Fix:** adopt Doc B's approach as the only one: a generator tool (rcgen) writes a persistent dev **CA + leaf with SANs `localhost`, IP `127.0.0.1`, IP `::1`** to one agreed gitignored path (pick one â€” Doc B says `dev/certs/`, Doc C says `infrastructure/docker/certs/`); server loads leaf+key; client appends the CA to its `RootCertStore`. Verify rcgen emits IP SANs (it does via `SanType::IpAddress`) since the client may dial `127.0.0.1`, and rustls 0.23 validates IP SANs.

**3. [BLOCKER as written] Doc B's `axum-server 0.7 "tls-rustls"` pulls `rustls/aws-lc-rs`** (verified on 0.8.0; same on 0.7), which builds `aws-lc-sys` â†’ requires CMake + NASM â€” and worse, the cmake it would find on this PATH is the **MSYS2 MinGW cmake**, guaranteeing a broken MSVC build. **Fix:** don't use axum-server at all. Serve axum (REST + `ws` upgrade) behind a small hand-rolled `tokio-rustls` accept loop in `network-core` (Doc B already names this as its own fallback; make it the primary). If axum-server is kept anyway, it must be `tls-rustls-no-provider` + a ring-built `ServerConfig`. This also resolves the Doc A vs Doc B WSS-stack split (raw tokio-tungstenite server vs axum `ws`): use **axum 0.8 `ws`** (which now uses tokio-tungstenite 0.29, matching Doc A's pin) over the tokio-rustls listener â€” one HTTP stack, one port, no duplicate tungstenite in the tree.

**4. [LIKELY-PAIN] reqwest is a one-`cargo update` aws-lc-rs trap.** Doc C's `reqwest 0.12` + `rustls-tls-manual-roots` is verified ring-backed â€” but current reqwest is **0.13.4 whose rustls default is aws-lc-rs**. Any casual bump (or a Tauri plugin pulling reqwest 0.13) drags aws-lc-sys/NASM/CMake into the client build. **Fix:** pin `reqwest = "0.12"` with `default-features = false`, document why; if 0.13 is ever adopted, use its `rustls-no-provider` feature plus an installed ring provider. Add the guard from #5.

**5. [LIKELY-PAIN] rustls "no process-level CryptoProvider" runtime panic + provider drift.** With six TLS consumers (quinn, tokio-rustls, sqlx, async-nats, tungstenite, reqwest), if feature unification ever enables both `ring` and `aws-lc-rs` on rustls, every `ClientConfig::builder()`/`ServerConfig::builder()` **panics at runtime**. **Fix (both):** (a) call `rustls::crypto::ring::default_provider().install_default()` as the first line of every bin (monolith, each service bin, Tauri host) and/or use `builder_with_provider` in `network-core`; (b) add a checked gate to `just check`/CI: `cargo tree -i aws-lc-sys` must return "nothing depends on it" â€” this single command also enforces #3, #4, and protects against future deps (e.g. jsonwebtoken 9 is ring-backed â€” verified-safe â€” but do not bump to 10.x without re-checking its crypto backend).

**6. [LIKELY-PAIN] MSYS2 on PATH will poison any -sys crate that shells out.** `C:\msys64\mingw64\bin\cmake.exe` and `C:\msys64\usr\bin\perl.exe` precede nothing-else on this machine. The ring-only policy removes the main consumer, but any future dep invoking cmake gets MinGW-configured output against an MSVC toolchain with bizarre errors. **Fix:** note in `docs/getting-started.md`; optionally have `scripts/bootstrap.ps1` warn when `(Get-Command cmake).Source` is under `msys64`. Pair with the #5 CI gate.

**7. [LIKELY-PAIN] Crate-version skew between the three docs must be reconciled to one workspace table.** Concretely: prost **0.14** (A; B/C say 0.13 â€” 0.14.4 is current and `protoc_executable` lives there), tokio-tungstenite **0.29** (A; B/C say 0.26 â€” 0.29 also matches axum 0.8.9's ws), async-nats **0.49** (A; B says 0.38 â€” stale), redis **1.x** (A; B says 0.29 â€” stale; note `redis::aio` API moved, code from current docs), metrics-exporter-prometheus 0.17 vs 0.16 â€” pick 0.17. Also Doc B names a nonexistent tokio-tungstenite feature `rustls-tls` (real names: `rustls-tls-webpki-roots` / `-native-roots`); the client should instead pass `Connector::Rustls(config)` and needs no roots feature. **Fix:** Doc A's Â§2.1 `workspace.dependencies` is the single source of truth; Docs B/C inherit `{ workspace = true }`; client workspace mirrors the same pins.

**8. [LIKELY-PAIN] sqlx: right plan, three sharp edges.** (a) Pin **sqlx-cli to the same 0.8.x minor** and install it as `cargo install sqlx-cli --version "^0.8" --locked --no-default-features --features rustls,postgres` â€” the default install compiles mysql/sqlite/native-tls for nothing; a 0.9 cli against a 0.8 workspace is undefined behavior for the `.sqlx` cache. (b) `cargo sqlx prepare --workspace -- --all-targets` needs `DATABASE_URL` set *and* migrations applied â€” `just sqlx-prepare` must depend on `db-setup`. (c) Doc A's justfile recipes reference `$env:DATABASE_URL` but nothing loads `.env` into just recipes â€” add `set dotenv-load := true` at the top of the justfile (sqlx macros and sqlx-cli read `.env` themselves; `just` does not by default).

**9. [LIKELY-PAIN] Tauri 2 runtime vs network-core spawning.** No fundamental conflict (Tauri 2's `tauri::async_runtime` *is* a tokio runtime, and tokio-tungstenite/quinn run happily on it), but `GatewayClient::connect` "spawns the driver task" â€” `tokio::spawn` from the Tauri **setup hook or main thread panics** ("no reactor running"). **Fix:** make `connect` take a `tokio::runtime::Handle` explicitly (cleanest for a library crate), or standardize on `tauri::async_runtime::spawn`, or create one tokio Runtime first and hand it to Tauri via `tauri::async_runtime::set(handle)`. Decide once in `state.rs`/`lib.rs`; do not leave it implicit. (Doc C's `panic = "abort"` experiment: keep it flagged exactly as the doc does â€” try late, not load-bearing.)

**10. [LIKELY-PAIN] "One shared `Arc<rustls::ClientConfig>` for quinn + tungstenite + reqwest" (Doc C Â§1.7) cannot be literal.** QUIC **requires** ALPN `dice/1` on its config (handshake fails without it); reqwest wants h2/http1.1 ALPN; WSS wants http/1.1. Additionally `reqwest::ClientBuilder::use_preconfigured_tls` requires the *exact* rustls version reqwest links â€” a silent-skew trap. **Fix:** share one `RootCertStore` + provider; build three small configs from it (clone-and-set-ALPN); for reqwest use the public `add_root_certificate(...)` + `tls_built_in_root_certs(false)` API, never preconfigured TLS. Note `quinn::crypto::rustls::QuicClientConfig::try_from` then works with the ring provider as designed (0-RTT/early-data stays off â€” matches all three docs).

**11. [LIKELY-PAIN] Cross-doc structural skew (non-wire) â€” pick one of each before scaffolding:** monolith crate (`services/monolith`/`dice-monolith` in A vs `services/dice-server` in B); config (A's env-only loader in `dice-common` vs B's figment `crates/config` â€” pick A's, it's smaller and B's `DICE__*` layering can be added later); task runner (A's justfile vs B's xtask â€” keep the justfile; the only thing that genuinely needs Rust is cert/JWT-key generation, which fits as a tiny `cargo run -p dice-devtool` recipe rather than a whole xtask convention); **permissions bit tables differ materially** (A: `ADMINISTRATOR = 1<<63`, default `131`; B: `ADMINISTRATOR = 1<<6`, default `35` â€” and B's migration hardcodes `DEFAULT 35` in SQL).
